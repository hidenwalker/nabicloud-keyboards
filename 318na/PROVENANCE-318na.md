# 세벌식 3-18Na (318Na) — 독립 재유도 로그 (PROVENANCE)

본 문서는 나비구름(NabiCloud)의 318Na 자판 데이터(`keyboards/318na.xml`)와 엔진 측
종성-순환 전처리(`shared/engine/nabicloud/engine/engine.c`)가 **공개 동작 명세에서
독립 재유도**되었음을, 출처 → 사실 → 데이터의 대응으로 기록한다.

> ★**[2026-06-25 V2 확장 XML 이관]** 318Na 는 산들바람 V2 데이터-구동 엔진으로
> 이관됐다. 배포본 `keyboards/318na.xml` 은 이제 **V2 확장 XML**(`<jaso engine="v2"
> role-flip="jongcycle">`, `jaso_layout_318na.c` C 정본을 `jaso_xml_dump`
> 로 직렬화한 것 — 손편집 금지·재생성)이고, 런타임은 구엔진의 `jongseong_cycle`
> 전처리가 아니라 `JASO_ROLE_JONGCYCLE` 역할(데이터-구동)로 동작한다. 위에서 기록한
> **libhangul-body 원본**(`<map>`/`<combination>`/`<multikey>` 표현, 이 문서가 재유도를
> 서술하는 그 데이터)은 구엔진 동작 오라클로 **`oracle/318na.xml`** 에 보존한다
> (배포·스캔 경로 아님). 검증 체인: C layout ≡ oracle(`selftest_jaso_318na`,
> oracle-anchored, 73 checks) · V2 XML ≡ C layout(`selftest_jaso_roundtrip`, C≡XML)
> ⇒ V2 XML ≡ oracle. 구엔진 골든(`golden_318na`)은 CI 게이트에서 은퇴(재현기로 보존).

## 1. 작성 방식 선언 (독립 재유도)

- navilera/NavilIME·navilera/318Na_HangulKeyboard·pat-al/Online-HanGeul-IME(OHI)
  등 GPL 계열 코드의 **표현**(자료구조 레이아웃·식별자·함수 본문·직렬화 포맷·주석
  문장)은 **일절 복제하지 않았다.**
- 위 코드와 기존 나비구름 전사본(`318na.xml`/`layout-318na.{c,h}`)은 **동작
  오라클(behavior oracle)** 로만 사용했다 — 즉 "어떤 키시퀀스가 어떤 음절을
  만드는가"라는 *동작 사실*의 교차검증·회귀 비교용이며, 그 *표현*을 베끼지 않았다.
- 검증 기준은 **기능 동치**(키시퀀스 → 유니코드 정규화 음절 회귀,
  `tests/golden_318na.c`)이지 byte-동치가 아니다. byte-동치를 목표로 삼으면
  독립 재유도 원칙 위반이므로, 데이터 직렬화는 의도적으로 전사본과 byte-비동치다
  (`git diff`로 확인: 섹션 구조·항목 순서·주석을 모두 자체 표현으로 재작성).

## 3. 사실 → 데이터 재유도

### 3.1 키맵 (`<map>`)
- **초·중성**: 표준 두벌식 QWERTY 배열을 그대로 둔다(S1·S2 "두벌식 무수정").
  → q ㅂ / w ㅈ / e ㄷ / r ㄱ / t ㅅ / a ㅁ / s ㄴ / d ㅇ / f ㄹ / g ㅎ /
    z ㅋ / x ㅌ / c ㅊ / v ㅍ, 쌍자음 Shift; 중성 y ㅛ / u ㅕ / i ㅑ / o ㅐ /
    p ㅔ / h ㅗ / j ㅓ / k ㅏ / l ㅣ / b ㅠ / m ㅡ / n ㅜ.
- **종성 직접 글쇠**: 세미콜론 `;` → 1차 종성 ㅈ(S1·S2 저빈도 종성 배치). 겹받침
  Shift 글쇠 H/I/J/K/M/B → ㄶ/ㅀ/ㅄ/ㄺ/ㄻ/ㄼ(S1 2019 겹받침 보강 + 표준 겹받침
  집합). 자모를 두지 않는 글쇠는 ASCII 통과.

### 3.2 결합 (`<combination>`)
- **초성 쌍자음·중성 이중모음**: 표준 한글 오토마타와 동일(S1 "초중성 두벌식
  무수정"이므로 결합도 표준).
- **종성**: ① 같은 종성키 반복 → 1차/2차 단일종성 토글(ㄱ↔ㄲ, ㅅ↔ㅆ, ㅈ↔ㅊ);
  ② 서로 다른 종성키 → 표준 겹받침 결속(ㄱㅅ=ㄳ, ㄴㅈ=ㄵ, ㄴㅎ=ㄶ, ㄹ+{ㄱㅁㅂㅅㅌㅍㅎ}=
  ㄺㄻㄼㄽㄾㄿㅀ, ㅂㅅ=ㅄ). 모두 한글 표준 겹받침 집합(S3 분해식이 함의).

### 3.3 종성 순환 (`<multikey>`)
- 오른손 모음 글쇠가 종성으로 재해석될 때의 후보쌍(S1 표 정본, S4 교차검증):
  ㅜ→ㄹ/ㅋ, ㅠ→ㅂ/ㅍ, ㅕ→ㅅ/ㅆ, ㅛ→ㄱ/ㄲ, ㅑ→ㅇ/ㅎ, ㅗ→ㄴ/ㅌ, ㅡ→ㅁ/ㄷ.
  `first`=재해석 trigger(중성), `second`=1차 종성, `third`=2차 종성(획 증가형).
- **gap 결정**: ㅗ키 2차 종성은 명세 gap(navilera 정본=ㄴ 단독 vs OHI=ㄴ/ㅌ)이었다.
  동작 오라클(기존 데이터·OHI 2:1)과 회귀 일관성을 위해 **ㄴ/ㅌ**를 채택했고,
  `tests/golden_318na.c`의 회귀가 이 선택을 고정한다.

## 4. 엔진 측 재유도 (`nabicloud_engine_jongseong_cycle`)

- navilera 함수 포팅을 폐기하고, **범용 데이터주도 전처리**로 독립 재작성했다.
  순환 테이블은 데이터(`hic->keyboard->multikey`)이며, 함수는 단일 모드
  JONGSEONG_CYCLE + **2후보 고정**(primary/secondary)으로 버퍼 상태를 읽어 종성을
  결정한다(N후보 일반화는 의도적으로 하지 않음 — 자판 설계가 plain↔획증가 쌍뿐).
- 동작 사실로 보존한 상태전이(공개 명세·오라클 회귀):
  1) 중성만 있을 때 순환키 → 1차 종성(초성 없으면 단독 종성 표시 위해 중성 드롭);
  2) 종성이 이미 있을 때 → 1차 후보 결합 시도 → 실패 시 2차 후보 결합 → 둘 다
     불가면 기존 종성 제거 후 2차 후보 단독(도깨비불 토글).
  ★두 단계는 같은 호출에서 **둘 다 적용될 수 있는 순차 가드**(else-if 아님)다 —
  jaso 버퍼가 일시적으로 중성+종성을 동시에 가질 수 있어 순서를 보존해야 한다.
- 게이트는 `nabicloud_engine_preprocess` 하나로 단일화했다(production dispatcher와
  Sandeulbaram 하니스가 동일 진입점 사용). 순환 테이블이 없는 자판(전 빌트인 26종)은
  identity 통과라 byte-불변이다.

  rky→각, rk→가, skh→난, dlu→잇(1차 ㅅ), dluu→있(2차 ㅆ), kh→ㄴ 단독,
  dl;→잊(종성 ㅈ), dl;;→잋(종성 ㅊ), djn→얼, djny→얽(ㄹ+ㄱ→ㄺ; 중성·종성 동시 비0),
